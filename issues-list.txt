### 6. Implement Request Signature Verification (Ed25519)
**Labels:** backend, security, critical
**Description:**
**Problem:**
- No request signature verification between frontend and backend (HIGH-2 finding)
- Vulnerable to CSRF attacks and request tampering
- Any client could submit arbitrary distribution requests
- Currently only CORS protects endpoints

**Solution:**
- Implement Ed25519 signing for all write operations (POST/PUT/DELETE)
- Frontend signs requests with private key, backend verifies signature
- Add signature header validation middleware
- Create signing utility in frontend to sign JSON payloads
- Add integration tests for signature verification

**Acceptance Criteria:**
- [ ] Signature middleware validates all write requests
- [ ] Frontend signs initialize, distribute, and admin requests
- [ ] Requests without valid signature rejected with 401
- [ ] Signature validation tests added (5+ test cases)
- [ ] Documentation updated with signing flow
- [ ] Backwards compatibility maintained for existing clients (optional grace period)

**Note for Contributors:**
- Use @stellar/stellar-sdk Ed25519 utilities
- Include nonce in signature to prevent replay attacks
- Add timestamp validation (no signatures older than 5 minutes)
- Mainnet blocker - coordinate with team before deployment

### 17. Implement Commit-Reveal Scheme for Initialization
**Labels:** smart-contract, security, critical
**Description:**
**Problem:**
- No front-running protection on initialize (HIGH-3 finding)
- Unauthorized admin could initialize before real admin (race condition)
- No commit-reveal prevents malicious initialization
- Collaborator list could be intercepted and modified

**Solution:**
- Implement commit-reveal pattern:
  - commit_initialize(collaborators_hash, shares_hash) - stores hash
  - reveal_initialize(collaborators, shares) - verifies hash and initializes
- Add delay between commit and reveal (1 block)
- Revert reveal if commit hash doesn't match
- Add nonce to prevent replay

**Acceptance Criteria:**
- [ ] commit_initialize(hash1, hash2) stores hashes
- [ ] eveal_initialize(collabs, shares) verifies and initializes
- [ ] Reveal fails if hash mismatch
- [ ] Reveal must be after commit + 1 block
- [ ] 6+ commit-reveal scenario tests
- [ ] Frontend guides user through 2-step process

**Note for Contributors:**
- Uses Soroban's current block number for delay
- Update frontend to show commit pending state
- Backend must coordinate 2-step flow
- Mainnet blocker
